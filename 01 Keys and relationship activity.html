SQLite Activity: Keys and Relationships

Important: Committing Changes in DB Browser for SQLite

  • After executing each SQL statement or group of related statements, you must manually save your changes.
  • To save changes, click the "Write Changes" button (floppy disk icon) in the toolbar, or go to "File" > "Write Changes" in the top menu.
  • If you don't write changes, your work will only exist in memory and will be lost if you close the application.
  • Get in the habit of writing changes after each major step in this activity.

Objective

In this activity, you will create a small database for a library system using SQLite and DB Browser. You'll practice implementing primary keys, foreign keys, and relationships between tables.

Instructions

1. Create a new database

  1. Open DB Browser for SQLite
  2. Click "New Database" in the toolbar (icon looks like a blank page) or go to File > New Database
  3. Navigate to your desired save location
  4. Name your database file "library.db"
  5. Click "Save"
  6. If prompted to create a table, click "Cancel"

2. Create tables

For each table:

  1. Click on the "Execute SQL" tab
  2. Paste the CREATE TABLE statement into the SQL editor
  3. Click the "Execute SQL" button (play icon)
  4. Click the "Write Changes" button (floppy disk icon) in the toolbar

Authors Table

CREATE TABLE Authors (
    AuthorID INTEGER PRIMARY KEY AUTOINCREMENT,
    FirstName TEXT NOT NULL,
    LastName TEXT NOT NULL
);

Books Table

CREATE TABLE Books (
    BookID INTEGER PRIMARY KEY AUTOINCREMENT,
    Title TEXT NOT NULL,
    PublicationYear INTEGER,
    AuthorID INTEGER,
    FOREIGN KEY (AuthorID) REFERENCES Authors(AuthorID)
);

Patrons Table

CREATE TABLE Patrons (
    PatronID INTEGER PRIMARY KEY AUTOINCREMENT,
    FirstName TEXT NOT NULL,
    LastName TEXT NOT NULL,
    Email TEXT UNIQUE
);

Loans Table

CREATE TABLE Loans (
    LoanID INTEGER PRIMARY KEY AUTOINCREMENT,
    BookID INTEGER,
    PatronID INTEGER,
    LoanDate DATE NOT NULL,
    ReturnDate DATE,
    FOREIGN KEY (BookID) REFERENCES Books(BookID),
    FOREIGN KEY (PatronID) REFERENCES Patrons(PatronID)
);

3. Create an index

An index helps to speed up searches on a specific column. We'll create an index on the Email column in the Patrons table.

  1. Click on the "Execute SQL" tab
  2. We'll use the following SQL command to create the index: 
    CREATE INDEX idx_patron_email ON Patrons(Email);
  3. Let's break down this command:
    • CREATE INDEX: This tells SQLite we want to create a new index
    • idx_patron_email: This is the name we're giving to our index. It's a good practice to use a descriptive name
    • ON Patrons(Email): This specifies that the index is on the Email column of the Patrons table
  4. Copy and paste the CREATE INDEX statement into the SQL editor
  5. Click the "Execute SQL" button
  6. Click the "Write Changes" button to save this modification to your database
  7. To verify the index was created:
    • Go to the "Browse Data" tab
    • Select "Patrons" from the table dropdown
    • Click on the "Indexes" sub-tab
    • You should see your new index listed

4. Add a CHECK constraint using a trigger

We'll create a trigger that enforces the condition that the PublicationYear in the Books table is not beyond 2024.

  • Click on the "Execute SQL" tab
  • Use the following SQL command to create the trigger:
CREATE TRIGGER check_publication_year
BEFORE INSERT ON Books
BEGIN
    SELECT CASE 
        WHEN NEW.PublicationYear > 2024
        THEN RAISE(ABORT, 'Publication year cannot be later than 2024')
    END;
END;

Let's break down this command:

  • CREATE TRIGGER check_publication_year: This creates a new trigger named "check_publication_year"
  • BEFORE INSERT ON Books: This specifies that the trigger should run before any INSERT operation on the Books table
  • The SELECT CASE statement checks if the new PublicationYear is greater than 2024
  • If it is, it raises an error and aborts the INSERT operation
  • Copy and paste the entire CREATE TRIGGER statement into the SQL editor
  • Click the "Execute SQL" button
  • Click the "Write Changes" button to save this modification to your database

To test the trigger, try inserting a book with a year beyond 2031:

INSERT INTO Books (Title, PublicationYear, AuthorID)
VALUES ('Far Future Book', 2031, 1);

This should fail due to the trigger.

Now try inserting a book with a valid year:

INSERT INTO Books (Title, PublicationYear, AuthorID)
VALUES ('Future Book', 2029, 1);

This should succeed.

And of course, current and past years should also work:

INSERT INTO Books (Title, PublicationYear, AuthorID)
VALUES ('Current Book', 2023, 1);

This should also succeed.

4. Add a CHECK constraint

We'll add a CHECK constraint to ensure the PublicationYear in the Books table is not beyond 2024. We'll recreate the table with the constraint, working within the existing transaction.

  • Click on the "Execute SQL" tab
  • Use the following SQL commands to recreate the Books table with the constraint:
-- Create a temporary table with the new structure
CREATE TABLE Books_temp (
    BookID INTEGER PRIMARY KEY AUTOINCREMENT,
    Title TEXT NOT NULL,
    PublicationYear INTEGER CHECK (PublicationYear <= 2024),
    AuthorID INTEGER,
    FOREIGN KEY (AuthorID) REFERENCES Authors(AuthorID)
);

-- Copy data from the old table to the new one, adjusting any years beyond 2024
INSERT INTO Books_temp (BookID, Title, PublicationYear, AuthorID)
SELECT 
    BookID, 
    Title, 
    CASE 
        WHEN PublicationYear > 2024 THEN 2024
        ELSE PublicationYear
    END,
    AuthorID
FROM Books;

-- Drop the old table
DROP TABLE Books;

-- Rename the new table to the original name
ALTER TABLE Books_temp RENAME TO Books;

-- Optional: Display any books that had their publication year adjusted
SELECT * FROM Books WHERE PublicationYear = 2024;

Let's break down these commands:

  • We create a new temporary table with the desired structure and constraint (PublicationYear <= 2024).
  • We copy data from the old table, adjusting any years beyond 2024 to be exactly 2024.
  • We drop the old table and rename the new one to 'Books'.
  • Optionally, we display any books that now have 2024 as their publication year.
  • Copy and paste the entire SQL statement block into the SQL editor
  • Click the "Execute SQL" button
  • Click the "Write Changes" button to save this modification to your database

To test the new constraint, try inserting a book with a year beyond 2024:

INSERT INTO Books (Title, PublicationYear, AuthorID)
VALUES ('Future Book', 2025, 1);

This should fail due to the CHECK constraint.

Now try inserting a book with a valid year:

INSERT INTO Books (Title, PublicationYear, AuthorID)
VALUES ('Current Book', 2024, 1);

This should succeed.

And of course, past years should also work:

INSERT INTO Books (Title, PublicationYear, AuthorID)
VALUES ('Old Book', 2000, 1);

This should also succeed.
